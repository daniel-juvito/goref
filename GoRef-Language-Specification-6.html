Language Specification

GoRef

VERSION 1.0 · DRAFT · TRANSPILES TO GO

A Go-compatible language that replaces the dual-purpose pointer (*T) with two explicit constructs — one for mutation, one for optionality. Reference-like types collapse automatically. Non-nil interfaces are statically enforced. Every GoRef program compiles to idiomatic, readable Go.

§ 01

Introduction

GoRef is a statically typed, compiled language that transpiles to idiomatic Go. It is syntactically and semantically identical to Go in almost every way, with one deliberate departure: the pointer type (*T) is replaced by two distinct, purpose-built constructs.

In Go, a pointer (*T) serves two fundamentally different purposes:

  • Mutation via reference — passing a value so a callee can modify the caller's copy.
  • Optionality / nil — expressing that a value may be absent.

Conflating these two concerns forces programmers to always ask: "Is this pointer here because the value might be nil, or because I need to mutate it?" GoRef eliminates that ambiguity by splitting the concept cleanly.

&T
Reference Type

Always holds a valid T, never nil. Enables mutation across call boundaries.

→ Go: *T
?T
Nilable Type

May be absent (nil). Must be nil-checked before use.

→ Go: *T
&?T
Reference to Nilable

A non-nil reference to a slot whose value may be nil. Combines both concerns explicitly.

→ Go: *T
◆
Design Philosophy

GoRef does not add a runtime. Every GoRef program compiles to valid, readable Go code. The transpiler is the only additional tool in the chain. All three types — &T, ?T, and &?T — transpile to *T in Go; the distinctions are enforced statically by the transpiler. Neither * nor & ever appear in GoRef source — intent is expressed entirely through type annotations. &T on reference-like types collapses silently to T, making generics work naturally. Plain interface parameters without ? are statically guaranteed non-nil.

§ 02

Goals & Non-Goals

Goals

  • Be a strict superset of Go syntax (modulo pointer syntax).
  • Produce idiomatic, human-readable Go output.
  • Make nil-safety statically verifiable at transpile time.
  • Eliminate implicit nil dereferences with zero runtime overhead.
  • Preserve all Go semantics: goroutines, channels, interfaces, embedding.
  • Implicit referencing — callers never write & when passing to &T parameters.

Non-Goals

  • GoRef is not a new runtime or VM.
  • GoRef does not change Go's memory model or garbage collector.
  • GoRef does not introduce ownership or borrow-checking.
  • GoRef does not change performance characteristics vs plain Go.
§ 03

Type System

3.1 Reference Type  &T

A reference type &T is a non-nil alias to a T value. It transpiles to *T in Go, but the transpiler statically guarantees that no nil is ever assigned to it.

GoRef 
func increment(n &int) {
    n += 1   // no * needed — transpiler emits *n += 1
}

x := 10
increment(x)   // x is implicitly referenced; caller writes no &
// x is now 11

Rules for reference types

  • &T variables must be initialized at declaration time.
  • Assigning nil to &T is a transpile-time error.
  • Passing a value of type T to a parameter of type &T is implicit — the transpiler inserts &.
  • Assigning through a reference (v = newValue) is implicit — the transpiler emits *v = newValue.
  • Field access and method calls are always transparent: v.Field, v.Method() — no * needed.
  • The * operator does not exist in GoRef source. It only appears in generated Go.
  • Returning &T from a function that may return nil is a transpile-time error.

3.2 Nilable Type  ?T

A nilable type ?T represents a value that may be either a valid T or nil. It transpiles to *T in Go.

GoRef 
func findUser(id int) ?User {
    // ...may return nil
}

u := findUser(42)

// u.Name  <- transpiler error: ?User used without nil check

if u != nil {
    fmt.Println(u.Name)  // OK inside the guard
}

Rules for nilable types

  • Any field, variable, or return value of type ?T must be guarded by a nil check before any member access, method call, or dereference.
  • nil is a valid literal for ?T.
  • Assigning ?T directly to T (without unwrapping) is a transpile-time error.
  • The guard may be an if v != nil block, an if v, ok := expr; ok pattern, or a type-switch.

3.3 Reference to Nilable  &?T

The type &?T is a reference to a nilable slot — the slot itself always exists (non-nil reference), but the value inside may be nil. It transpiles to *T in Go, identical to ?T at the wire level. The distinction is purely in GoRef's type system: the transpiler enforces that the reference itself cannot be nil, while the value it points to may be.

GoRef → Go 
// GoRef
func clearUser(u &?User) {
    u = nil   // transpiler emits *u = nil
}

var current ?User = getUser()
clearUser(current)   // implicitly referenced

// Generated Go — &?T transpiles to *T, same as ?T
func clearUser(u *User) {
    *u = nil   // transpiler inserted *
}
current := getUser()
clearUser(&current)  // transpiler inserted &

Rules for &?T

  • The reference itself is always non-nil (outer & guarantee).
  • The inner ?T still requires a nil-guard before access — or is auto-promoted inside one.
  • Transpiles to *T in Go — identical wire type to ?T, non-nil guarantee enforced statically.

Invalid Type Combinations

??T (double nilable) is invalid (GR009) — nil is already the sole absent value for ?T, nesting it adds nothing. ?&T where T is a plain value type is invalid (GR012) — &T already guarantees non-nil, making the outer ? a contradiction. However, ?&T where T is a reference-like type is valid — &T collapses to T first, so ?&T becomes ?T. &&T written explicitly also collapses to &T via the same rule.

3.5 Reference-Like Types & & Collapsing

Go's slices, maps, channels, functions, and interfaces already carry an internal pointer. Applying & to them is not an error — instead, &T collapses silently when T is already reference-like. The transpiler simply omits the * in the generated Go. This makes generics work naturally without any special constraints.

Collapse Rule

&T where T is a slice, map, channel, function, or interface collapses to T — the transpiler emits no * and no implicit & at call sites. The implicit-referencing and auto-deref rules are simply skipped. ?T remains valid and meaningful for all these types.

3.6 Zero Value Rule for Reference-Like Types

The zero value of slices, maps, channels, functions, and interfaces in Go is nil. In GoRef, declaring one of these types without initialization is ambiguous — it is unclear whether nil is intentional or an oversight. GoRef resolves this by requiring an explicit choice at declaration time. This rule applies to both variable declarations and struct field definitions.

✕
GR011 — Uninitialized Reference-Like Type

Declaring a reference-like type without initialization and without ? is a transpile error. You must either mark it nilable with ?T or provide an initial value.

GoRef 
// ERROR GR011: reference-like type declared without initialization or ?
var s []int
var m map[string]int
var c chan int
var f func() error
var w io.Writer

// ✅ Option 1 — explicitly nilable
var s ?[]int
var m ?map[string]int
var c ?chan int
var f ?func() error
var w ?io.Writer

// ✅ Option 2 — initialized, never nil
var s []int = []int{}
var m map[string]int = make(map[string]int)
var c chan int = make(chan int)
s := []int{}          // short form also valid

// ERROR GR011: struct fields also require ? or initialization
type Foo struct {
    Data []int          // ❌ must be ?[]int or always initialized
    Name string         // ✅ string zero value is "" not nil
}

// ✅ Struct field options
type Foo struct {
    Data ?[]int         // ✅ explicitly nilable
    Name string         // ✅ fine
}

// ERROR GR011: struct literal must initialize all non-nilable reference-like fields
f := Foo{}              // ❌ Data []int left uninitialized
f := Foo{Data: nil}    // ❌ nil not allowed for []int field
f := Foo{Data: []int{}} // ✅

Why This Matters

Unintentionally nil maps and channels are one of the most common sources of runtime panics in Go. Writing to a nil map panics. Sending to a nil channel blocks forever. GoRef makes the nil-or-not decision explicit at the declaration site, eliminating this entire class of bugs at transpile time.

3.7 Embedded Types

GoRef supports ?T and &T on embedded struct fields. Both transpile to *T anonymous embed in Go — the distinction is enforced statically by the transpiler. The same nil-safety rules that apply to regular fields apply to embedded types.

GoRef
type Bar struct {
    ?Foo    // nilable embed
    &Baz    // non-nil embed
}

b := Bar{Baz: Baz{}}
// ?Foo omitted — ok, it's nilable

b.BazMethod()    // ✅ &Baz always non-nil

b.FooMethod()    // ❌ GR002 — ?Foo needs nil-guard

if b.Foo != nil {
    b.FooMethod() // ✅ auto-promoted
}
Generated Go
type Bar struct {
    *Foo    // ?Foo → *Foo embed
    *Baz    // &Baz → *Baz embed
}

b := Bar{Baz: &Baz{}}
// Foo is nil — zero value of *Foo

b.BazMethod()    // safe

// transpiler inserted nil-guard:
if b.Foo != nil {
    b.FooMethod()
}

3.8 Variadic Parameters

Nilability on variadic parameters applies to each element, not the slice. The variadic slice itself is auto-created by Go from the call arguments and is never nil. ?T on a variadic parameter simply transpiles each element to *T.

GoRef
// each element may be nil
func log(msgs ...?string) {
    for _, m := range msgs {
        if m != nil {
            fmt.Println(m)
        }
    }
}

// elements are plain values
func log(msgs ...string) {
    for _, m := range msgs {
        fmt.Println(m)  // always safe
    }
}
Generated Go
// each element is *string
func log(msgs ...*string) {
    for _, m := range msgs {
        if m != nil {
            fmt.Println(*m)
        }
    }
}

// unchanged
func log(msgs ...string) {
    for _, m := range msgs {
        fmt.Println(m)
    }
}
§ 04

Implicit Referencing

When a function parameter is declared as &T, the caller does not need to write &. The transpiler inserts it automatically. This is the single largest ergonomic win over raw Go.

GoRef — source
func scale(v &Vec3, factor float64) {
    v.X *= factor   // field access — transparent
    v.Y *= factor
    v.Z *= factor
}

func reset(n &int) {
    n = 0           // assign through ref — no * needed
}

pos := Vec3{1.0, 2.0, 3.0}
scale(pos, 2.0)   // no & needed
Generated Go
func scale(v *Vec3, factor float64) {
    v.X *= factor   // Go auto-derefs fields
    v.Y *= factor
    v.Z *= factor
}

func reset(n *int) {
    *n = 0          // transpiler inserted *
}

pos := Vec3{1.0, 2.0, 3.0}
scale(&pos, 2.0)  // transpiler inserted &
◆
Important — No Double & Insertion

The transpiler only inserts & at call sites when the argument is a plain T. If the variable is already &T (declared as y &int := x), no additional & is inserted — the transpiler passes y directly as the pointer. Neither * nor & ever appear in GoRef source — the transpiler always inserts both as needed.

§ 05

Nil Safety Rules

5.1 What Counts as a Nil Guard

The transpiler performs flow-sensitive nil analysis. Inside a guarded scope, ?T is automatically promoted — no unwrap operator needed. The following constructs lift a ?T into a non-nil scope:

Valid Nil Guards 
// 1. Simple if-check
if v != nil { use(v) }

// 2. Early return pattern
if v == nil { return }
use(v)  // safe after the guard

// 3. Short-variable form
if v, ok := maybeGet(); ok { use(v) }

// 4. Switch
switch {
case v != nil:
    use(v)
}

5.2 Violations — Transpile Errors

Transpile Errors 
u := findUser(42)   // ?User

// ERROR: field access on ?User without nil check
fmt.Println(u.Name)

// ERROR: passing ?User where T is expected
processUser(u)   // processUser takes User, not ?User

// ERROR: assigning ?User to plain User
var plain User = u

The transpiler reports all nil-safety violations before emitting any Go code. It never produces partially valid output.

5.3 Flow-Sensitive Auto-Promotion

Inside a nil-guarded scope, the transpiler automatically promotes ?T to a non-nil value without any extra syntax. The guard already proved non-nil, so assignment is safe and the transpiler handles the unwrap silently.

GoRef — auto-promotion inside guard 
u := findUser(42)   // ?User

if u != nil {
    name := u.Name   // OK — ?User auto-promoted, no ! needed
}

for node.Next != nil {
    node = node.Next   // ?Node auto-promoted — loop condition proved non-nil
}

if node == nil { return }
name := node.Name   // OK — early return proved non-nil

5.4 Unwrapping Operator !

The v! postfix operator is for the rare case where the programmer can prove non-nil through logic the transpiler cannot see — such as external invariants or index-based lookups. It asserts non-nil at runtime and panics with a descriptive message if nil is encountered.

GoRef — source
u := findUser(42)   // ?User
// programmer knows id 42 always exists
name := u!.Name
Generated Go
if u == nil {
    panic("nil dereference: ?User at main.gr:12")
}
name := u.Name
⚠
Use Sparingly

Prefer nil guards — they keep safety at transpile time. Use ! only when the transpiler cannot see the proof of non-nil. Inside any nil-guarded scope, ! is never needed.

§ 07

Declarations & Syntax

7.1 Short Variable Declarations

GoRef 
x := 42             // int
z ?int := nil       // nilable int, nil
w ?int := 7         // nilable int, value 7

y := &x              // ✅ y is &int — explicit reference binding
y = 100             // transpiler emits *y = 100, modifies x

z := *x              // ERROR: dereference * not available in GoRef source
◆
& in GoRef source — one exception

Neither & nor * ever appear in GoRef source. Reference intent is expressed through the type annotation: y &int := x. The transpiler sees that y is &int and x is int, and emits y := &x in Go automatically — exactly like passing x to an &int parameter.

GoRef
x := 42
y := &x        // y is &int
y = 100        // assign through reference
fmt.Println(x) // prints 100
Generated Go
x := 42
y := &x        // *int
*y = 100       // transpiler inserted *
fmt.Println(x) // prints 100

7.2 Multiple Return Values with Nilables

GoRef — idiomatic error-or-value pattern 
func readConfig(path string) (?Config, error) {
    data, err := os.ReadFile(path)
    if err != nil {
        return nil, err
    }
    var cfg Config
    // ... parse ...
    return cfg, nil   // cfg implicitly referenced to ?Config
}

cfg, err := readConfig("app.toml")
if err != nil { log.Fatal(err) }
if cfg == nil { log.Fatal("no config") }
fmt.Println(cfg.Port)  // safe: nil-checked above

7.3 Interface Types

In Go, nothing prevents passing a nil interface to a function that accepts io.Writer — it compiles but may panic at runtime. GoRef fixes this by letting you express nilability intent directly in the interface parameter type. A plain interface type (no ?) is guaranteed non-nil at the call site. Since interfaces are reference-like, &Writer collapses to Writer — so non-nil interfaces are expressed simply as Writer.

GoRefGenerated GoNil allowed?
Writerio.Writer❌ No — transpiler rejects nil at call site (GR001)
?Writerio.Writer✅ Yes — must nil-guard before use (GR002 otherwise)
GoRef
type Writer interface {
    Write([]byte) (int, error)
}

// non-nil guaranteed — no guard needed
func send(w Writer, data []byte) {
    w.Write(data)   // always safe
}

// nilable — must guard
func trySend(w ?Writer, data []byte) {
    if w == nil { return }
    w.Write(data)   // safe after guard
}

send(nil, data)     // ❌ GR001 — Writer is non-nil
trySend(nil, data)  // ✅ ?Writer accepts nil
Generated Go
type Writer interface {
    Write([]byte) (int, error)
}

// same signature — guarantee is static
func send(w Writer, data []byte) {
    w.Write(data)
}

func trySend(w Writer, data []byte) {
    if w == nil { return }
    w.Write(data)
}

// transpiler rejected send(nil) at compile time
trySend(nil, data)
◆
Same Wire Type, Different Static Guarantee

Both Writer and ?Writer transpile to io.Writer in Go — identical at runtime. The difference is purely static: the transpiler rejects nil at call sites for Writer and requires nil-guards inside functions for ?Writer. This closes a long-standing nil interface pitfall in Go with zero runtime cost.

7.4 Error Handling

The error interface is a special case in GoRef. Because returning nil for error is universal Go convention (meaning "no error"), error in a return type is implicitly treated as ?error by the transpiler. You never need to write ?error explicitly.

GoRef — error is implicitly ?error in return position 
func foo() (int, error) {
    return 42, nil              // ✅ nil error = success
    return 0, MyError{Code: 404} // ✅ concrete error value
}

// interfaces that embed error are NOT implicitly nilable
type DetailedError interface {
    error             // embeds error
    Code() int
    Context() map[string]string
}

func bar() (int, DetailedError) {
    return 0, MyDetailedError{} // ✅
}

func baz() (int, ?DetailedError) {
    return 42, nil              // ✅ explicit ? needed for non-error interfaces
}

7.5 Nil Pointer in Interface Trap

One of Go's most notorious bugs is returning a nil pointer wrapped in an interface — the interface is non-nil even though the pointer inside is nil. GoRef's nil-safety system prevents this naturally.

Go — classic bug
func getError() error {
    var err *MyError = nil
    return err
    // NOT nil error!
    // interface wraps nil pointer
    // err != nil is TRUE at call site
}
GoRef — caught at transpile time
func getError() error {
    e ?MyError := nil
    return e       // ❌ GR003 — ?MyError
                   // assigned to error
                   // without nil guard

    // must be explicit:
    return nil     // ✅ nil error
}
✓
Nil Pointer in Interface — Eliminated

Because ?MyError requires a nil guard before being assigned to error, the transpiler catches the nil-pointer-in-interface trap at transpile time. To return "no error" you must return nil directly — never a ?T that might be nil.

7.6 Interfaces Embedding error

Interfaces that embed error do not inherit its implicit nilability. The ?error special case applies only to the built-in error type in return positions — not to any interface that happens to embed it. Use ?DetailedError explicitly if nilability is needed.

§ 08

Control Flow & Nil Propagation

8.1 Flow-Sensitive Analysis

The transpiler tracks nil-status through all control flow paths including if/else branches, early returns, and for loops. It uses a conservative lattice: if any path reaching a point may leave a ?T unchecked, the use is flagged.

GoRef — branch-local guards 
u := findUser(id)  // ?User

if someCondition {
    if u == nil { return }
    // u is safe HERE (in this branch)
} else {
    // u is NOT safe HERE — guard only in the if-branch
}

// u is NOT safe here — guard was branch-local
fmt.Println(u!.Name)  // must use ! or re-check

8.2 For Range over Nilable Slices

A ?[]T slice must be nil-checked before ranging over it.

GoRef 
items ?[]string = getItems()

// ERROR: ranging over ?[]string without nil check
for _, s := range items { ... }

// OK
if items != nil {
    for _, s := range items { fmt.Println(s) }
}

GoRef 
type UserRepo interface {

    Save(u &User) error

func greetUser(repo UserRepo, id int) error {
    u, err := repo.FindByID(id)
    if err != nil {
        return fmt.Errorf("db error: %w", err)
    }
    if u == nil {
        return fmt.Errorf("user %d not found", id)
    }
    fmt.Printf("Hello, %s!\n", u.Name)  // safe
    u.LastSeen = time.Now()
    return repo.Save(u)  // implicit &u
}

§ 12

Safe Navigation Operator ?.

The ?. operator allows chaining field access and method calls through a nilable value without explicit nil guards. If the receiver is nil, the entire chain short-circuits to nil. The result type is always ?T.

GoRef
u := findUser(42)   // ?User

// without ?.
var street ?string
if u != nil {
    if u.Address != nil {
        street = u.Address.Street
    }
}

// with ?. — equivalent
street := u?.Address?.Street
Generated Go
u := findUser(42)

var street *string
if u != nil {
    if u.Address != nil {
        _s := u.Address.Street
        street = &_s
    }
}



// transpiler expands ?. chain

12.1 Rules

  • The result of v?.Field is always ?T where T is the type of Field.
  • Chains short-circuit at the first nil: u?.Address?.Street returns nil if either u or u.Address is nil.
  • Method calls are supported: u?.FullName() returns ?string.
  • The result of ?. is a ?T and must be nil-guarded before further use — unless chained with another ?..
  • ?. cannot be used on non-nilable types — the transpiler emits a warning and simplifies to plain ..
GoRef — chaining examples 
// field chain
street := u?.Address?.Street   // ?string

// method call
name := u?.FullName()          // ?string

// mixed chain
zip := u?.Address?.ZipCode()   // ?string

// using result
if street != nil {
    fmt.Println(street)        // auto-promoted
}

// with unwrap (programmer asserts non-nil)
fmt.Println(u?.Address?.Street!)
◆
Relationship to Auto-Promotion

?. and auto-promotion are complementary. Auto-promotion is for when you want to explicitly guard and then work safely inside the block. ?. is for when you want to traverse a chain and deal with the result once at the end — without nesting multiple guards.

§ 13

Transpiler Development Roadmap

The following features are planned for the GoRef transpiler implementation, ordered by development priority.

Phase 1 — Core

1.1 Lexer Extend Go lexer with ? token and ?. operator.
1.2 Parser Extend Go AST with &T, ?T, &?T type nodes and ?. expression nodes.
1.3 Type Checker Resolve all type annotations. Build reference/nilable maps. Enforce collapse rules for reference-like types. Validate &T, ?T, &?T, GR009, GR012.
1.4 Nil Flow Analysis Flow-sensitive analysis over SSA-like IR. Track nil-status through all branches, early returns, and loops. Auto-promote ?T inside guarded scopes. Flag GR002, GR003, GR006, GR008.
1.5 Code Generation Emit Go source. Insert * for reference assignments. Insert & at call sites. Handle collapse. Expand ?. chains. Run gofmt on output.
1.6 GR011 Checker Validate all variable declarations, struct field definitions, embedded types, and struct literals for uninitialized reference-like types.

Phase 2 — Tooling

2.1 CLI goref build, goref transpile, goref vet, goref watch commands.
2.2 Source Maps Map generated Go lines back to GoRef source for debugger integration.
2.3 Error Messages Human-readable errors with file, line, column, type context, and suggested fixes for all GR001–GR012 codes.
2.4 LSP Server GoRef Language Server for IDE integration — hover types, inline nil-safety warnings, go-to-definition, autocomplete.

Phase 3 — Interoperability

3.1 Go Import Annotations Annotation files (.gra) that describe the nilability of imported Go package APIs — marking which return values are nilable and which parameters are non-nil.
3.2 stdlib Annotations Bundled .gra annotation files for Go's standard library covering common packages: os, net/http, io, sync, context.
3.3 Incremental Transpilation Cache transpiled output and only re-transpile changed files and their dependents.
